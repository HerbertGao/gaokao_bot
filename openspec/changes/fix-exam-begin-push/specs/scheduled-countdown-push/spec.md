## 新增需求

### 需求:定时推送在开考那一刻发送专用文案

系统在定时推送任务中，当某场考试处于「开考那一刻」（当前北京时间在开考时刻之后、且在开考后第一分钟之内）时，必须向所有发送目标推送一条专用文案「{exam}开始了！」，其中 `{exam}` 为考试全称。由于定时任务在整点触发、取到的当前时间略晚于开考整点，开考整点的那次触发会命中该判定；该判定一年仅命中一次（仅限开考后第一分钟内的那一个 tick）。当结果集中同时存在多场考试时，系统必须对每场考试按其各自开考时刻独立判定，命中者各自推送（与既有倒计时逐场推送一致）。

该专用文案禁止改动 `GetCountDownString` 的输出，因此 Inline Query、Bot 命令等非定时推送路径在考试进行中时仍返回「{exam}正在进行中！」，行为不变。

#### 场景:开考整点触发推送

- **当** 定时任务在开考整点（如 `2026-06-07 09:00`）触发，且取到的当前北京时间落在开考时刻之后、开考后一分钟之内
- **那么** 系统必须向所有发送目标推送「{exam}开始了！」
- **并且** 该文案必须使用考试全称（`ExamDesc`），而非倒计时模板渲染结果

#### 场景:整秒边界不命中（漏发）

- **当** 某次触发取到的当前时间恰好等于开考时刻（`now == begin`，整秒边界，概率可忽略）
- **那么** 系统既不命中开考那一刻判定（判定要求严格「在开考之后」），「≤24 小时」倒计时分支也因要求严格「大于 0」而不命中 → 该次触发不推送任何消息（即该 tick 漏发，与 cron 错过窗口同类）
- **并且** 系统禁止在此情形下误发「正在进行中！」或倒计时文案、亦不得崩溃

#### 场景:开考一分钟后不再发送开考文案

- **当** 定时任务触发时，当前时间已超过开考后一分钟（仍在考试进行中）
- **那么** 系统禁止再推送「开始了！」文案
- **并且** 此时是否推送及推送内容由其它既有发送规则决定（不在本需求范围内）

### 需求:定时推送沿用既有发送频率与判定

系统的定时推送在开考那一刻之外的发送规则必须保持不变：距考试 24 小时以上时仅在每日 9:00 推送倒计时；距考试 24 小时以内（且尚未开考）时每小时推送倒计时；倒计时文案、考试进行中/已结束判定与时间标准化逻辑均不变。开考那一刻的判定必须先于「≤24 小时每小时」判定生效，以保证开考整点发送的是「开始了！」而非倒计时或「正在进行中」。

注：在定时推送路径下，考试一旦结束即被 `GetExamsInRange` 移出结果集（其过滤上界 `exam_year_end_date` 等于 `exam_end_date`），故 `GetCountDownString` 的「已结束」分支在定时推送中实际不可达——这是既有行为，本次不改变。

#### 场景:距考试超过 24 小时

- **当** 当前距某场考试开考超过 24 小时，且当前为每日 9:00（含 9:00–9:01 容差窗口）
- **那么** 系统必须推送该考试的倒计时文案
- **并且** 在非 9:00 的其它整点不得推送

#### 场景:距考试 24 小时以内且尚未开考

- **当** 当前距某场考试开考在 24 小时以内且尚未到开考时刻
- **那么** 系统必须在每个整点推送该考试的倒计时文案

#### 场景:开考瞬间优先于倒计时判定

- **当** 开考整点触发，既满足「开考那一刻」又处于「≤24 小时」窗口边界
- **那么** 系统必须按开考那一刻处理，推送「{exam}开始了！」
- **并且** 禁止退化为倒计时或「正在进行中」文案
